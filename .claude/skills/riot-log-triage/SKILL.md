# Riot Log Triage

对 RIOT 项目的日志类问题执行分阶段排障：先确认日志产物、时间口径和文件路由；再用首轮日志搜索建立订单/车辆/状态候选；然后读代码推导期望下一步和关键门禁；最后回日志验证为什么没有走到下一步。对车辆/订单执行问题，先建立可证伪的“问题时间点快照”；缺订单号时先找 `order_candidates`，不要停在追问，也不要把未闭合证据包装成高置信根因。

## Select the Project

先识别项目，再只读取对应 reference：

- `allspark` / `riot3` / `RIOT3`：读取 [references/riot3-allspark.md](references/riot3-allspark.md)
- `riot-standalone` / `riot2` / `RIOT2`：读取 [references/riot2-standalone.md](references/riot2-standalone.md)
- `fms-java` / `riot1` / `RIOT1` / `fms`：读取 [references/riot1-fms.md](references/riot1-fms.md)

项目未明确时，不要假设。先说明当前无法锁定日志栈和配置入口，再只问 1 个确认问题，例如“这是 RIOT1 / RIOT2 / RIOT3 哪一套？”

## Choose the Triage Mode

- 简单问题走 `会话态`：
  - 用户已经给出明确项目、版本、时间、日志片段
  - 日志文件少，证据链短
  - 用户只要一个快速结论或下一步建议
- 复杂问题走 `结构化状态流`：
  - 有附件、图片、导出包或多文件日志
  - 版本信息不明或可能不准
  - 问题时间与日志时间可能不匹配
  - 需要多轮补料
  - 需要跨多个日志文件、多个模块建立证据链
  - 用户后续可能要求正式报告、`5 why` 或根因分析
- 进入复杂模式时，读取 [references/workflow-state.md](references/workflow-state.md)。
- 无论哪种模式，主线程都只保留压缩后的状态、关键证据和代码引用，不要把大段原始日志持续塞进主上下文。

## Apply the Hard Evidence Rules

- 先判断问题是否属于`订单 / 车辆任务执行链路`。
- 只要问题涉及下单、派单、任务下发、车辆执行、取消、完成、卡死、状态流转异常等，都默认按`订单 / 车辆任务执行问题`处理；在 RIOT 里，车辆任务执行通常依托订单，不要把车辆任务执行问题和订单链路拆开分析。
- 只要属于`订单 / 车辆任务执行问题`，继续读取 [references/order-execution-triage.md](references/order-execution-triage.md)；把重点放在“还原车辆/订单运行过程 -> 找出问题时间点状态和流程 -> 根据代码判断期望下一步 -> 回日志验证为什么没有走到下一步”。
- 对`订单 / 车辆任务执行问题`，`车辆名称`是全程主锚点；如果同一车辆在问题窗口内牵出多个订单，不要省略，必须把订单候选显式列出来，再判断哪个订单与主问题闭合。
- 对`订单 / 车辆任务执行问题`，强证据链按 `订单ID -> 车辆名称 -> 时间` 闭合。`订单ID` 是最强证据；缺少订单ID时，不要把这类问题的结论包装成高置信。
- 只有明确不属于`订单 / 车辆任务执行链路`的问题，才可以不强求 `订单ID`；这时至少要让 `时间` 和 `车辆名称` 能闭合到同一现场。
- `RIOT3 / allspark` 原始日志时间按 `UTC+0` 记录。对照现场时间时，必须先换算到项目所在时区：国内项目统一按 `UTC+8`，其他项目按项目所在地时区。输出证据时，优先同时标注“日志原始时间”和“换算后的现场时间”。
- `RIOT2 / riot-standalone` 和 `RIOT1 / fms` 以服务器时间为准，不要套用 `RIOT3` 的 `UTC+0` 规则；先确认服务器时区，再判断时间窗是否对齐。

## Separate Discovery From Verification

把日志搜索分成两类，不要混用规则：

- `首轮探索 / order_discovery`：目标是找 `order_candidates`、关键状态、当前动作和可能的流程门禁。缺订单号时，用`车辆名称 + 时间窗 + 问题业务门禁词`做宽召回，业务词之间可以 OR，按目标文件和 `term_priorities` 排序；这轮可以 `anchor_match_mode=prefer`，但必须在输出里标注哪些命中还没有订单闭合。
- `二轮及以后 / verification`：目标是验证某个具体假设。拿到订单候选后，搜索应收紧为`订单ID + 车辆名称 + 时间窗 + 门禁/代码关键词`；这轮默认要求锚点闭合，不能只靠泛词或相似报错。
- `结论分级`：探索轮只能给部分结论或候选假设；只有验证轮把订单、车辆、时间、关键日志和代码门禁闭合后，才能给高置信根因。
- `DSL 口径`：DSL 是给人和工具看的检索意图。首轮可以是 `AG0019 OR ChangeMapRequest OR CrossMapManager` 这类宽召回；验证轮应变成 `AG0019 AND 358208 AND (ChangeMapRequest OR vehicleProcState)` 这类强约束。

## Run the Workflow

### 1. Inspect the delivered log artifact first

- 先判断当前拿到的是哪一种：
  - 原始日志目录
  - UI 或接口导出的整包
  - 二次打包后的附件
  - 截图里截取出来的局部日志
- 先做文件层盘点，不要急着读业务日志正文：
  - 顶层目录结构
  - 文件命名规则
  - 是否按模块/日期/小时/主机/类型分组
  - 是否已经二次压缩
  - 压缩链是否为 `.zip`、`.tar.gz`、单文件 `.gz`、分卷或混合格式
- 如果是“系统导出日志”得到的包，优先搜索项目里的导出实现类，弄清楚：
  - 导出时是按什么规则聚合文件
  - 导出后最终产物是什么格式
  - 哪些日志会被纳入、哪些会被排除
  - 时间范围是按天、按小时，还是按文件修改时间筛选
- 只有明确了打包和压缩规则，才决定解压方式和后续筛选方式。

### 2. Check completeness and alignment

- 先判断关键信息是否完整：
  - 项目
  - 版本/分支/构建包
  - 问题描述
  - 问题模块
  - 问题时间
  - 项目所在时区或服务器时区
  - 日志时间格式和时区
  - 日志包是否完整
- 先判断是否存在错位：
  - 问题时间和日志时间是否同一时区
  - `RIOT3` 是否已经把 `UTC+0` 日志时间换算成项目现场时区
  - `RIOT2 / FMS` 是否已经按服务器时间理解问题时间窗
  - 问题发生时间是否落在当前日志包覆盖范围内
  - 问题描述模块与当前日志文件类型是否匹配
  - 用户给的截图/片段是否只是一小段上下文，无法代表完整日志
- 如果发现“信息不完整”或“时间/模块错位”，先把它当作阻塞项，而不是直接推根因。

### 3. Frame the incident

- 提取项目、环境、版本/分支、现象、期望行为、实际行为、首次时间、影响范围、是否可复现。
- 区分问题类型：`bug candidate`、`现场问题`、`配置/部署问题`、`数据问题`、`误操作`、`证据不足`。
- 列出已有证据：日志片段、完整日志文件、异常堆栈、截图、附件、配置文件、订单号、任务号、车辆名称、车辆号、设备号、线程名、主机名。
- 先把`主问题`写成一句话，并持续保留在状态里；对订单/车辆执行问题，优先写成“为什么这辆车在这个时间点没有继续执行/没有下发下一段动作”。
- 对`订单 / 车辆任务执行问题`，先建立“问题时间点快照”。首轮快照允许是 `partial`：先填已经确定的时间、车辆、候选订单、候选流程和待验证项；不要因为缺订单号或状态字段就停止搜索。最终快照至少包含：
  - 车辆系统状态，例如 `SystemState`
  - 车辆执行状态，例如 `vehicleProcState`
  - 当前子任务 / 当前动作 / 当前请求
  - 所处流程门禁，例如切图监听、回调校验、锁路、过电梯、等待资源
  - 理论上的下一步动作是什么，谁负责下发
- 如果这些项还没有闭合，不要急着把“没有规划路径”“HttpAct 卡住”“定位异常”“状态抖动”包装成主因；先把缺口写入 `missing_items` 或下一轮 `focus_question`。

### 4. Inspect logging config and routing

- 在搞清楚导出产物之后，再去读项目的 logging 配置：
  - `logback.xml`
  - `logback-file.xml`
  - `log4j2.xml`
  - 相关 include/appender/rolling policy 文件
- 重点确认：
  - 当前日志根目录
  - appender 对应的文件名模式
  - 是否按天/小时滚动
  - 是否 gzip 压缩
  - logger 是否 `additivity=false`
  - 当前故障更可能落在哪类文件，而不是默认主日志

### 5. Build the evidence pack

- 先标准化所有硬标识：精确时间、异常类名、logger 名称、订单/任务/车辆名称/车辆号/设备/request ID、容器或主机名。
- 强证据链优先锚点固定为：`订单ID -> 车辆名称 -> 时间`。
- 对`订单 / 车辆任务执行问题`，验证主因前必须收敛到同一 `订单ID`，再检查车辆名称和时间是否闭合；首轮如果还没有订单号，应明确这是 `order_discovery`，不能只靠模糊时间窗或相似报错替代订单证据。
- 对明确的非订单链路问题，至少确认车辆名称和时间与现场一致；如果两者对不上，先视为证据链未闭合。
- 对`订单 / 车辆任务执行问题`，证据包里必须显式放入一份 `incident snapshot`，并标注 `partial` 或 `confirmed`：
  - 问题时间点车辆状态是什么
  - 问题时间点流程走到哪一步
  - 当时正在等待什么条件
  - 理论下一步动作是什么
  - 当前哪条代码判断直接拦住了下一步
- 把现象分层：`主因候选`、`触发条件`、`伴随现象`、`干扰项`。只有能直接解释“为什么没继续下发下一步动作”的，才能放进主因候选。
- 如果用户说“看下附件 / 看下图片”，先检查已经给出的内容，不要直接反问。
- 如果当前项目存在本地 `log-analysis` skill 或宿主系统提供等价批量搜索机制，优先把“日志文件路由定位”或“批量日志筛选”交给它；否则使用本 skill 自带的 `search_worker.py`。主线程负责最终综合结论。
- 如果当前输入只有媒体占位信息，没有真正的图片或文件正文，明确指出“证据当前不可机读”，再追问最小补料，例如原始截图文字、原始异常堆栈或日志附件。

### 6. Locate the concrete classes and business path

- 先搜“导出日志 / 下载日志 / 打包日志”的控制器、service、presenter、工具类，确认日志包不是被二次改写过的。
- 在首轮日志已经确定候选时间线、订单候选或关键门禁后，再搜实际业务异常相关的源码：异常类、日志文本、相关业务链路、关键模块、入口方法。只有源码搜索成本很低且能直接生成首轮关键词时，才可以提前做轻量代码词提取。
- 去看具体 logger 所在类，而不是只看报错文案。
- 如果一个问题跨多层实现，先画出最短链路：入口接口 -> service -> domain/engine -> logger 类。
- 分析执行过程前必须先读懂执行链路；没有链路复盘时，只能说“命中到片段”，不能直接说“已经解释过程”。
- 执行链路至少要回答：
  - 入口动作是谁触发的，例如订单阶段、子任务、设备回调或监听器。
  - 当前状态/阶段是谁写入，谁读取，日志里的状态名对应哪个代码枚举或字段。
  - 请求下发、回调成功、状态校验、下一步动作之间的先后关系是什么。
  - 正常路径下一步应该调用哪个类/方法，现场证据显示它是否执行。
  - 异常路径是抛异常、返回失败、提前 return、挂起等待，还是被并发状态覆盖。
- 对“为什么没继续下发下一步动作”这类问题，优先找真正的决策门禁：
  - 谁设置了当前状态
  - 谁读取了当前状态
  - 哪个类/方法决定“继续下发 / 提前返回 / 挂起等待”
  - 哪个条件分支最直接解释现场不动
- 代码分析必须服务于日志证据、首轮候选和时间线，不要从代码结构自由发散：
  - 先按日志时间排序还原 `T1 -> T2 -> T3`，再读取对应 logger 类。
  - 读取类时优先找“日志文案所在方法 -> 状态读写 -> 下一步动作下发/提前返回”的最短路径。
  - 没有在日志时间线上出现的代码路径只能作为待验证候选，不能覆盖已有时间顺序。

### 7. Build the call chain and keyword package

- 从问题描述和首轮日志候选先梳理最小调用链，再回到代码验证：
  - 涉及模块
  - 入口接口/入口任务
  - 中间 service / domain / engine
  - 关键 logger 类
  - 关键异常类
- 对订单/车辆执行问题，调用链优先画成“状态设置方 -> 状态读取方 -> 下一步动作门禁”。
- 下一轮搜索词必须从执行链路节点提取，而不是只从报错文案扩展：
  - 入口节点：订单阶段、子任务类型、请求对象。
  - 状态节点：状态枚举、阶段名、状态字段、状态更新方法。
  - 决策节点：门禁判断方法、提前返回日志、失败枚举。
  - 输出节点：下一步请求、回调类、后续动作方法。
  - 异常节点：异常类、错误码、失败结果构造方法。
- 关键词提取顺序固定为：执行链路/调用链 -> 关键类/方法 -> 关键日志文案 -> 下一轮检索词。不要跳过调用链直接从日志泛词扩展。
- 从关键类提取关键日志时，保留“类名/方法名/日志文案/异常词”的绑定关系，例如 `CrossMapManager.crossMapSuccessAction -> 切换地图成功,车辆执行状态不符合`。
- 只保留“能帮助搜索日志”的关键类，不要把整条调用链全部塞进主上下文。
- 代码分析产出的关键词必须结构化，避免模型凭感觉扩词：
  - `core_terms`：类名、方法名、核心请求/响应类型，例如 `CrossMapManager`、`doCrossMap`、`ChangeMapRequest`。
  - `exception_terms`：异常类、错误码、失败枚举，例如 `AGV_CHANGE_MAP_TIME_OUT_ERROR`。
  - `log_message_terms`：源码里的关键日志文案，例如“切换地图成功,车辆执行状态不符合”。
  - `stage_terms`：状态、阶段、门禁条件，例如 `IN_CHANGE_MAP`、`ASSIGN_NEXT_SUB_TASK`。
  - `term_priorities`：按证据价值排序，优先级来自“是否贴近日志时间线、是否对应决策门禁、是否同时解释车辆/订单状态”。
- 为每个假设或每类日志文件生成一个 `keyword package`，至少包含：
  - `include_terms`：订单号、车辆名称、车辆号、任务号、设备号、异常类、关键方法、关键日志文案
  - `exclude_terms`：已知噪音、无关模块、误导性公共报错
  - `target_files`：预计命中的日志文件
  - `time_window`：按问题时间推导出的最小搜索窗口
- 首轮 `keyword package` 的目标是探索，不是证明根因：
  - 有订单号：可以放入 `anchor_terms`，但仍要加入业务门禁词，避免只扫订单/车号。
  - 缺订单号：保留车辆锚点，同时加入从问题描述和轻量代码词提取出的 `gate_terms/core_terms/log_message_terms/stage_terms`；允许业务词 OR 召回，并用 `term_priorities` 排序。
  - 首轮必须写明 `focus_question`，例如“找出 AG0019 在问题窗口内的订单候选和切图后状态”。
- 验证轮 `keyword package` 必须从已命中的时间线和代码门禁生成；默认用 `订单ID + 车辆名称 + 时间窗` 收紧，并把探索轮噪音加入 `exclude_terms` 或降级为 `noise_candidates`。
- 下一轮 `keyword package` 必须保留车辆/订单锚点；代码关键词按 `core_terms / exception_terms / log_message_terms / stage_terms` 分组，并写入 `term_priorities`。
- 逐轮收敛时，每一轮只解决 1 个 focus question，例如：
  - 这一刻车处于什么状态
  - 这一刻卡在哪个流程
  - 哪个状态被谁改掉了
  - 哪个门禁阻止了后续移动下发
- 每一轮都明确：保留了哪些关键词、删除了哪些关键词、时间窗缩小到哪里、哪几个现象先降级为干扰项。
- `订单 / 车辆任务执行问题`的 `keyword package` 默认把 `订单ID` 放在第一优先级；缺少订单ID时，不要只靠大时间窗做全局模糊搜索。
- `time_window` 必须先按项目时区规则归一化：`RIOT3` 用“项目现场时区 <-> UTC+0”换算后的时间窗，`RIOT2 / FMS` 用服务器时间窗。
- 拿到新的证据或补料后，允许重新输出 `keyword package`，不要死守第一次生成的关键词。

### 8. Delegate heavy search outside the main context

- 多文件日志搜索、批量 grep/zgrep、滚动日志筛选、日志去噪，优先使用 `search_worker.py`。如果宿主系统明确提供项目本地 `log-analysis` skill、独立子 agent 或 Task 机制，也可以下沉给它们。
- 如果已经进入复杂模式，优先先初始化 `.triage/.../00-state.json`，再让搜索 worker 基于 `keyword package` 执行。
- 只要满足下面任一条件，就默认把搜索下沉到 `search_worker.py` 或宿主系统允许的批量搜索执行器：
  - 目标日志跨 3 个以上文件或跨多个滚动分片
  - 单轮命中结果明显超过主线程可直接消化的量
  - 需要 2 轮以上逐步缩小时间窗和关键词
  - 需要同时验证主因候选和干扰项
- 下沉执行器只接收最小必要输入：
  - 当前问题摘要
  - 本轮 focus question
  - 时间窗口
  - `keyword package`
  - 目标日志文件或目录
- 下沉执行器返回时只回传：
  - 命中的关键证据摘要
  - 关联文件路径和时间点
  - 精选的原始日志内容
  - 这些日志是如何匹配到的，例如命中关键词、时间窗、目标文件
  - 关键代码引用
  - 命中统计与未命中说明
  - 本轮新增/删掉了哪些关键词，哪些假设被保留、哪些被降级为干扰项
- 不要把大段原始搜索结果原样贴回主线程，否则会把上下文打爆。
- 如果搜索结果噪音太大，优先调整关键词、缩小时间窗、增加排除词，必要时明确提出“需要去噪”。

### 9. Search before asking

- 先查日志导出实现、日志配置、部署配置、启动脚本、环境参数。
- 再查源码：异常类、日志文本、相关业务链路、关键模块、入口方法。
- 再查结构化记忆：历史现场问题、已知坑、版本差异、已有结论。
- 优先按“模块 + logger + 时间窗 + 业务 ID”收敛，不要一上来全局模糊搜。
- 不要向用户询问你可以从代码、配置、日志或历史记忆中自己得到的信息。

### 10. Review the evidence chain and confidence

- 证据链至少检查这几层是否能闭合：
  - 现场现象
  - 命中的日志证据
  - 对应的调用链/关键类
  - 配置或代码解释
- 对`订单 / 车辆任务执行问题`，最终至少要能正面回答这 3 个问题：
  - 问题时间点车辆处于什么状态
  - 问题时间点处于什么流程/门禁
  - 为什么后续动作没有继续下发
- 如果日志涉及多个文件，明确写出“证据跨哪些文件拼起来”，不要假装单文件就能解释全部问题。
- 判断证据链是否完整：
  - 是否有关键时间点
  - `订单 / 车辆任务执行问题`是否命中 `订单ID`
  - 是否有匹配的车辆名称
  - 是否有命中的关键 logger 或异常
  - 是否能对应到代码里的关键类和方法
- 必须单独检查时序一致性：
  - 搜索结果的高分排序只代表阅读优先级；结论必须按 `timeline_hits` 或原始日志时间顺序复盘。
  - 如果一个高分 hit 发生在低分 hit 之后，不能把它写成前置原因，除非有明确的跨文件时间校准证据。
  - 没有时间戳的结构化 trace 或摘要只能辅助定位阶段，不能单独推翻有时间戳的业务日志。
  - 如果时间顺序存在缺口，先写成 `temporal_gaps`，再决定是否缩小时间窗或补搜。
- `订单 / 车辆任务执行问题`缺少 `订单ID` 时，证据链最多算 `partial`，不要直接给高置信根因。
- 如果时间或车辆名称对不上，优先判断为时区错位、对象错位或日志窗口选错。
- 证据链不完整时，先判断缺口来自：
  - 用户信息不够
  - 日志包不完整
  - 日志噪音过大
  - 服务侧没有把必要上下文带出来

### 11. Build hypotheses with confidence

- 同时保留 1 到 3 个假设即可，不要铺太开。
- 对每个假设都记录：
  - 支持证据
  - 反证或缺口
  - 哪个缺失事实会改变结论
- 对主假设，优先组织成这种句式：
  - “在 `T` 时，车辆 `V` 处于 `X` 状态、位于 `Y` 流程，理论下一步应该是 `Z`；但 `G` 类/方法读取到 `C` 条件后提前返回或挂起，所以后续动作没有下发。”
- 置信度分级：
  - `高`：日志、配置、代码路径基本指向同一解释，且没有重要反证；`订单 / 车辆任务执行问题`还需要 `订单ID`、车辆名称、时间基本闭合。
  - `中`：主假设已经形成，但还缺 1 到 2 个关键事实，例如车辆名称、关键时间点，或非订单问题的局部日志上下文。
  - `低`：证据不足、互相冲突，多个解释都成立，或者`订单 / 车辆任务执行问题`缺少 `订单ID`。
- 低置信判断不能包装成确定性根因。

### 12. Drive the human-in-the-loop loop

只有在“缺失的信息会直接影响结论”时才追问。

追问时遵守下面规则：

- 先说清楚已经确认了什么。
- 再只问最小阻塞问题，通常 1 个，最多给 3 个可选补料方向。
- 优先追要：
  - `订单 / 车辆任务执行问题`如果日志探索无法抽取或候选冲突，再要 `订单ID`
  - 精确故障时间和对应时区
  - 该时间窗附近的原始日志片段
  - 截图中的原始报错文字或完整堆栈
  - 车辆名称 / 车辆号 / 任务号 / 设备号
  - 部署版本、分支、构建包信息
- 不要笼统说“把所有日志都给我”。

好的追问模式：

`目前能确认 X，更像 Y，不像 Z。要把结论从中置信提升到高置信，还缺一个关键证据：请给我这次的订单号；如果暂时没有，请再补 2026-04-15 10:23（UTC+8）对应的 reservation.log 片段和车辆名称。`

### 13. Report brief progress during the loop

- 每轮只做简要汇报，不直接生成完整正式报告。
- 中间汇报最多包含：
  - 当前阶段
  - 主问题
  - 已确认的关键证据
  - 关键代码/类引用
  - 当前置信度
  - 缺失项
  - 下一步动作
- 如果子 agent 已经跑过搜索，主线程只汇总结果，不转储原始日志内容。

### 14. Generate the formal report only after user confirmation

- 用户没有明确确认前，只输出中间结论、证据链和补料请求。
- 只有用户明确表示“可以出正式报告 / 继续出 5 why / 给最终结论”时，才生成完整正式报告。
- 正式报告可以包含：
  - 最终结论
  - 置信度
  - 证据链
  - `5 why`
  - 根因分类（Bug / 配置 / 数据 / 部署 / 现场操作 / 证据不足）
  - 建议动作
  - 需要服务侧补强的能力

### 15. Return the result

对工作类问题，优先输出 `format=rich` 的结构化 JSON。

`format=rich` 可以用于中间结论、证据摘要和补料请求；它不等同于正式报告。用户未确认前，不要在 rich 输出里塞入完整 `5 why`、最终根因定稿或正式报告章节。

默认包含：

- `summary`：一句话结论 + 置信度
- `sections`：
  - `主问题直接回答`
  - `当前结论`
  - `证据`
  - `不确定点`
  - `建议动作`
- 如果有日志证据，`table` 默认增加 3 列：`日志原文`、`日志文件`、`匹配关键词`
- `日志原文`：放精选后的关键日志原文，优先用最能支撑结论的 1 行；必要时可补充时间点
- `日志文件`：放文件路径，必要时补充行号
- `匹配关键词`：放命中的关键词，多个关键词用逗号分隔
- 如果还要解释为什么命中，可额外增加 `匹配说明`，写明时间窗、目标文件或筛选条件
- 可选 `table`：例如 `检查项 / 结果 / 证据 / 日志原文 / 日志文件 / 匹配关键词`
- `fallback_text`：和结构化内容一致的纯文本兜底

如果当前仍被证据阻塞，输出“部分结论 + 最小补料问题”，不要假装已经定位到根因。

## Apply the Confidence Rubric

- `高置信`：给出明确结论和直接行动建议；`订单 / 车辆任务执行问题`需要 `订单ID`、车辆名称、时间已闭合。
- `中置信`：给出主结论，明确缺哪个证据，并在必要时只问 1 个问题。
- `低置信`：明确说明证据不足，列出最可能的 1 到 3 个解释，并指出最能区分它们的那一个补料；`订单 / 车辆任务执行问题`缺少 `订单ID` 时默认不高于低到中置信。

## Feed Back When the Workflow Is Blocked

把补强建议分成两条通道，不要混在一起：

- `主服务调整建议`：
  - 需要增加哪些入口字段，例如版本、构建包、环境、导出时间范围、原始文件清单
  - 是否需要在主服务里先做日志去噪
  - 是否需要在主服务里先做图片 OCR / 附件解压 / 文本抽取
  - 是否需要把“日志格式、时区、路由摘要”提前作为元信息给到分析链路
  - 是否需要把大规模日志搜索固定下沉到子 agent / 子 skill
- `用户补料建议`：
  - 还缺什么上下文
  - 还要补哪个时间窗、哪个文件、哪个 ID
  - 还要补哪个模块的原始描述
  - 补料后应重新生成哪一版 `keyword package`

## Keep the Conclusion Honest

- 明确区分三种话术：
  - “日志已经证明”
  - “代码路径支持”
  - “结合现场现象推断”
- 明确区分“直接原因”和“干扰项”；只有能直接解释“为什么没继续下发下一步动作”的，才放进直接原因。
- 如果问题更像非 bug，例如部署、配置、数据、上游依赖或操作方式问题，要直接说明。
- 只有“时间 + 车辆名称”匹配、但没有 `订单ID` 时，可以说明现场相关性，不能把`订单 / 车辆任务执行问题`的根因写成“已证实”。
- 如果代码证据和现场描述冲突，要明确哪一边更强以及原因。
- 如果问题涉及上线、调度安全、数据安全或不可逆操作，要明确写出“需要人工确认后再执行动作”。

## Use Example Triggers

- “帮我看 RIOT3 一个订单卡死的问题，附件里有日志和截图。”
- “RIOT2 现场报 `DuplicateKeyException`，这到底是代码 bug 还是脏数据？”
- “RIOT1 的监控日志一直刷 WARN，但现场没报障，给我一个结论和处理建议。”
