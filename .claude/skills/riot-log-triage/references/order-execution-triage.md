# Order / Vehicle Execution Triage

用于高频的`订单 / 车辆任务执行问题`。这类问题的核心不是“多搜一点日志”，而是先还原运行过程，再判断问题时间点为什么没有进入期望的下一步。

## Use This Reference

在下面场景优先读取本文件：

- 用户问题本质是“为什么车不动”“为什么没有继续下发下一段移动”“为什么订单卡在某一步”
- 涉及切图、过电梯、锁路、回调校验、资源等待、子任务并发完成、状态异常
- 需要回答“问题发生时车处于什么状态、订单处于什么步骤”

## Core Method

固定按这个顺序做：

1. 先做首轮探索：用车辆、时间窗和业务门禁词找 `order_candidates`、状态候选和关键时间点。
2. 再还原问题时间点前后车辆和订单的运行过程；如果订单号还没唯一，保留候选并继续收窄。
3. 再看代码，判断这一步理论上应该怎么继续执行。
4. 最后回日志验证：为什么没有走到理论下一步。

不要反过来一上来全局扫日志，也不要脱离问题时间点只看代码。

## Anchor Rule

这类问题全程以`车辆名称`作为主锚点。

- 先确认“这辆车在问题时间点前后做了什么”。
- 再确认“这辆车在这个窗口里对应的是哪一个订单 / 哪几个订单”。
- 如果同一车辆在问题窗口里牵出多个订单，不要直接挑一个最像的就往下推；必须显式列出订单候选，再判断哪个订单与主问题闭合。

简化理解：

- `车辆名称`：主锚点
- `订单ID`：强证据
- `时间`：约束窗口

## Step 1: Build the Incident Snapshot

先建立一个可继续搜索的快照。首轮允许 `partial`，先回答能从输入或首轮日志确定的问题，把缺口写入下一轮 `focus_question`：

- 问题时间点是哪一刻，时区是否已经对齐
- 车是谁，订单是谁，任务是谁
- 车辆系统状态是什么，例如 `SystemState`
- 车辆执行状态是什么，例如 `vehicleProcState`
- 当前动作 / 当前请求 / 当前子任务是什么
- 订单当时处于什么步骤
- 当时卡在哪个流程门禁，例如切图监听、回调校验、锁路、过电梯、等待资源
- 理论下一步动作是什么

如果还没有明确订单号，也先不要停。先围绕车辆名称、问题时间窗和业务门禁词把窗口内的 `order_candidates` 找出来，再继续缩小。

如果这些问题还答不完整，不要把“没有规划路径”“HttpAct 卡住”“定位异常”“状态抖动”直接写成主因；只能写成候选或干扰项。

## Step 2: Restore the Running Timeline

如果已经有唯一订单号，围绕 `订单ID + 车辆名称 + 时间窗` 建立一条最小时间线。缺订单号时，先用 `车辆名称 + 时间窗 + 业务门禁词` 建立探索时间线，并显式列出候选订单。时间线至少覆盖：

- 进入当前步骤之前的上一个关键动作
- 问题时间点前后状态变化
- 请求发送 / 完成 / 失败
- 子任务开始 / 完成 / 回退
- 关键门禁日志
- 问题之后系统是否继续推进、提前返回、重试或挂起

优先保留下面几类事件：

- 状态设置事件
- 状态读取事件
- 下一步动作的发送事件
- 决策失败 / 提前返回 / 挂起等待事件

如果时间线里已经能看到“状态被改掉 -> 门禁判断失败 -> 下一步没下发”，问题会很快收敛。

如果当前还没有唯一订单号，就先退一步，以 `车辆名称 + 时间窗 + 业务门禁词` 建立探索时间线，并在这条时间线里抽取 `order_candidates`。只有把订单候选列出来，后面代码分析和下一轮日志筛选才不会飘。

对 `RIOT2 / RIOT3` 的任务执行流程类问题，首轮日志优先看 `bootstrap`。原因很简单：状态机、任务推进、步骤切换、很多主流程门禁日志基本先落在这里。其他专项文件例如 `reservation`、`notify`、`mini_trace`、`mapf`、`DeadLockDetector`，是在 `bootstrap` 基础上继续补证据，不要反过来先从专项文件起手。

## Step 3: Use Code to Infer the Expected Next Step

带着时间线去看代码，不要脱离现场去做抽象阅读。重点找：

- 谁设置了当前状态
- 谁读取了当前状态
- 哪个类/方法决定“继续下发 / 提前返回 / 挂起等待”
- 当前步骤成功后，理论下一步应该是什么
- 哪个条件分支会阻止进入下一步

对订单运行类问题，优先画成这条链：

`状态设置方 -> 状态读取方 -> 决策门禁 -> 下一步动作发送方`

如果链路很长，也只保留最短关键链路，不要把整条调用栈全塞进主线程。

## Step 4: Derive Keywords from Code, Then Verify in Logs

代码不是为了替代日志，而是为了生成更准的搜索词。优先提取：

- 状态字段
- 关键类名 / 方法名
- 关键日志文案
- 当前动作名 / 子任务名 / 请求名
- 下一步动作名
- 可能导致提前返回的条件词

再回日志验证这几件事：

- 问题时间点是否命中同一订单、同一车辆、同一时间窗
- 状态是否真的在问题前被改掉
- 门禁判断是否真的命中
- 理论下一步是否真的没有下发
- 如果没有下发，是提前返回、挂起等待，还是转去别的步骤

## Keyword Package Rules

- 首轮探索包：车辆名称是主锚点，订单号未知时不要强行要求订单；加入能代表问题业务的门禁词，例如 `ChangeMapRequest`、`CrossMapManager`、`vehicleProcState`、`IN_CHANGE_MAP`、`MoveRequest`，并用 `term_priorities` 排序。
- 首轮 DSL 可以宽召回，例如 `("AG0019" OR "ChangeMapRequest" OR "CrossMapManager")`；命中后必须按时间线和订单候选人工复核。
- 验证包：一旦订单候选明确，改用 `("AG0019" AND "358208" AND ("ChangeMapRequest" OR "vehicleProcState"))` 这类强约束，验证具体门禁或状态变化。
- 每一轮都记录保留、增加、删除和降级的关键词；不要把探索轮噪音直接带进最终结论。

## Evidence Chain Standard

这类问题的证据链至少要闭合成：

`问题现象 -> 问题时间点状态/步骤 -> 关键门禁日志 -> 对应代码条件 -> 理论下一步 -> 实际为什么没走到下一步`

如果少了“理论下一步”这一环，分析很容易停留在现象层。

## Distinguish Cause from Noise

强制把结论拆成四层：

- `直接原因`：直接解释为什么没有继续执行下一步
- `触发条件`：把问题引爆的前置条件
- `伴随现象`：问题发生时一起出现，但不是主因
- `干扰项`：看起来很像原因，但解释不了“为什么没继续下发下一步”

示例：

- `直接原因`：门禁类读取到错误状态后提前返回
- `触发条件`：并发子任务完成把共享状态提前改掉
- `伴随现象`：状态抖动、定位重置、业务重试
- `干扰项`：某个宽泛报错、没有路径显示、非关键日志噪音

## Minimal Output

中间结论至少包含：

- `主问题`
- `问题时间点快照`
- `运行过程时间线`
- `期望下一步`
- `实际阻塞点`
- `当前置信度`
- `还缺什么证据`

如果暂时只能证明“运行过程卡在某个门禁”，就先停在这里，不要为了凑完整根因把干扰项抬成主因。
